# 页面设计文档模板 UIDesignTemplate

使用方式：阶段 3 基于 Figma 页面 Frame 生成 `docs/原型设计-Design/页面设计文档-UIDesign.md`。生成前先基于 `docs/原型设计-Design/视觉设计系统模板-DESIGNTemplate.md` 产出 `docs/原型设计-Design/视觉设计系统-DESIGN.md`。Lovart PSD/PNG 只作为备份参考；本文件只服务代码还原，不写作品集说明。

## 1. 设计稿索引

| 编号 | 页面 | 端 | Figma Frame | 备用 Lovart 稿 | 是否有效 | 导航规则 |
| --- | --- | --- | --- | --- | --- | --- |
| P01 |  | Web / 小程序 | Figma node 链接 | `design/lovart/pages/` | 有效 / 仅参考 / 缺失 | 显示 Tab / 返回页 / 固定底部按钮 |

无效设计稿包括：多屏拼贴、作品集总览、交互概览、独立弹层图、被裁切页面、底部 Tab 缺失页面。无效稿不能作为代码还原主依据。

Figma Frame 匹配规则：

- 用户只提供 Figma 文件链接或 Page 根节点链接时，先读取 Figma metadata，不要直接要求用户逐页复制 Frame 链接。
- 按 `ScreenInventory` 的页面编号、页面中文名、Lovart 文件名和 Figma Frame 名称建立匹配表，例如 `P01 登录注册`、`01-登录注册`、`登录注册` 都可视为同一候选。
- 优先读取当前 Page 的第一层 Frame 树；只有页面名匹配失败时，才少量下钻候选节点，不遍历无关图层。
- UI Design 必须记录匹配依据：`Figma Frame 链接/nodeId`、`匹配来源`、`置信度`。置信度可写：高 / 中 / 低。
- 只有同一页面出现多个候选、找不到候选或命名明显冲突时，才向用户索要具体 Frame 链接，并在阻塞项中列出需要确认的页面。

## 3. 导航与安全区

| 场景 | 规则 |
| --- | --- |
| 一级 Tab 页面 | 必须显示同一组底部 Tab，当前页高亮，内容区预留 Tab 高度 |
| 二级/详情/表单页面 | 默认顶部返回，不显示底部 Tab |
| 固定底部操作区 | 必须加 `env(safe-area-inset-bottom)`，内容区预留底部空间 |
| 微信小程序按钮 | 普通按钮优先用 `view/text + bindtap`，原生 `button` 只用于微信原生能力 |

验收规则：

- `已通过`：代码页面与 Figma Frame 主要视觉一致，只有可接受的小偏差。
- `待修复`：存在明显偏差，必须回到实现阶段修复。
- `阻塞`：缺设计稿、缺资源、无法截图或无法确认，应写清原因。

## 6. 开发注意事项

- 小程序不得用 Vant 默认样式替代设计稿；必须按本文件覆盖样式。
- Figma 中可识别图标先检查根目录 `assets/icons/`，再复制到对应端；无法直接取出的图片可先用稳定占位并写明阻塞。
- 截图对比能做就做；不能截图时，必须完成静态对照并在 GoalPlan 写明阻塞。
- 本文档未拆出的页面，不允许在第 4 步实现阶段直接标为已完成。
